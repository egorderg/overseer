## Context

The codebase currently operates without a standardized TypeScript pipeline, leaving many modules untyped and increasing runtime defect risk. This change introduces a project-level TypeScript foundation that works across React, Node.js, and Electron surfaces while allowing incremental migration from existing JavaScript. The design must integrate with current build/test tooling and avoid blocking ongoing feature delivery.

## Goals / Non-Goals

**Goals:**
- Establish a single TypeScript compiler configuration suitable for app, backend, and Electron processes.
- Enable incremental migration by allowing JavaScript and TypeScript to coexist during rollout.
- Add type-aware validation in local workflows and CI.
- Define shared core interfaces for module boundaries where runtime integration errors are common.

**Non-Goals:**
- Full migration of all JavaScript files to TypeScript in one release.
- Immediate adoption of maximum strictness flags that would break large areas without staged remediation.
- Replacing existing bundler/runtime architecture.

## Decisions

1. Introduce a base TypeScript config with environment-specific extensions.
   - Rationale: A base `tsconfig` keeps common compiler behavior consistent while allowing separate settings for renderer, main process, and tooling contexts.
   - Alternatives considered:
     - Single monolithic `tsconfig` for all targets: simpler initially, but harder to tune for mixed runtime environments.
     - Fully separate unrelated configs: flexible but encourages drift and inconsistent checks.

2. Use incremental migration mode (`allowJs` plus `checkJs` only where useful) before stricter enforcement.
   - Rationale: This permits immediate value from TypeScript without requiring a disruptive all-at-once rewrite.
   - Alternatives considered:
     - Hard cutover with strict compile gate: fastest purity, highest delivery risk.
     - No compile checks until full migration: low friction, but delays reliability gains.

3. Add explicit type-check commands to CI and local scripts.
   - Rationale: A dedicated type-check step catches integration errors independent of bundling and keeps feedback fast.
   - Alternatives considered:
     - Type-check only during full build: fewer steps, but slower failure feedback.
     - Editor-only typing: no CI guarantees and inconsistent team enforcement.

4. Prioritize typed contracts at boundaries (API payloads, IPC messages, shared config objects).
   - Rationale: Boundary typing yields the highest reduction in production issues for the least migration effort.
   - Alternatives considered:
     - Migrate leaf utility files first: easy wins but lower systemic impact.
     - Migrate UI-only first: improves component ergonomics but misses cross-process reliability risks.

## Risks / Trade-offs

- [Compiler noise from legacy JS] -> Mitigation: start with scoped checks, suppress only where justified, and tighten rules in phases.
- [Longer CI duration] -> Mitigation: use incremental compile caches where possible and keep type-check as a focused step.
- [Inconsistent typing patterns across teams] -> Mitigation: define minimal conventions for interfaces/types and enforce via lint/type rules.
- [Temporary dual-language complexity] -> Mitigation: track migration by area and prioritize high-risk modules first.

## Migration Plan

1. Add TypeScript dependencies, base config, and target-specific config extensions.
2. Introduce `typecheck` scripts and wire them into CI.
3. Convert high-risk integration boundaries (API/IPC/shared config) to typed contracts.
4. Gradually migrate modules by domain while maintaining app behavior parity.
5. Increase strictness in controlled phases once baseline issues are resolved.

Rollback strategy:
- Remove CI type-check gating and keep runtime build path unchanged if rollout blocks delivery.
- Retain non-invasive type annotations where helpful while deferring strict enforcement.

## Open Questions

- Which strictness flags should be enforced in phase one versus deferred (`noImplicitAny`, `strictNullChecks`, etc.)?
- Should emitted JavaScript be generated by `tsc` or remain delegated to existing bundler/transpiler flows?
- What is the preferred convention for shared domain types (central package vs feature-local ownership)?
